# 执行摘要

育见是一款面向家庭教育场景的智能理解系统。我们观察到：在 K12 阶段，大量家长并非缺少育儿建议，而是缺少对**自己孩子**在**自己家庭**中的处境理解——孩子为何拖延、为何对抗、为何沉默，往往与家庭互动方式、压力结构与沟通节奏相关，而非单一「不听话」或「不努力」所能解释。

通用大模型降低了对话成本，但难以持续记住一个具体家庭、串联多个真实生活片段，更难以把家长的焦虑、评价与孩子的事实区分开来。传统教育咨询虽专业，却频次低、难沉淀、难规模化。

育见提出并正在验证以下核心假设：

> **如果** AI 能够通过类咨询方式系统采集家庭场景上下文，将非结构化叙述转化为可推理的原子事实网络，并在此基础上构建持续演进的 FamilyModel（家庭认知模型）与孩子专属 SecondMe 档案，**那么** 家长将获得比通用问答更深、更贴、更可行动的家庭教育理解，并形成「越用越懂这个孩子」的长期闭环。

当前，育见 Web 主站已上线（https://yujian.yihe.site），完成四模块 Onboarding、日常流式交流、任务验证、沟通预演与画像沉淀等核心能力；项目处于**清华 iCenter 支持的公益试点与产品验证期**，在九州问学 30+ 支教支队及北京丰台区部分学校、社区开展真实场景测试。**尚未进入规模化收费阶段。**

我们寻求【待补全：融资轮次与金额】，主要用于模型与推理成本、产品工程（含小程序全量上线）、试点运营及合规安全建设，以完成从「公益验证」到「商业模式可验证」的关键跨越。

# 第一章 核心假设与问题定义

## 1.1 我们提出的三个核心假设

**假设一：家庭教育问题的本质，往往是「理解问题」而非「知识问题」。**

家长刷到的方法论往往正确，但无法解释「为什么我家孩子是这样」。作业拖延、手机冲突、沟通升级等现象，在不同家庭中对应不同的触发链条与互动循环。没有家庭级上下文，任何建议都容易退化为鸡汤。

**假设二：AI 要进入家庭教育，必须从「单次问答」升级为「长期建模」。**

单次对话可以安慰，却无法积累。可规模化的家庭 AI 必须能够：记住本家庭结构与原话、区分家长情绪与孩子事实、在新信息进入时修正旧判断（含反证），并把这些理解沉淀为可检索、可推理的结构。

**假设三：家长愿意投入叙述成本，前提是「被理解」而非「被考核」。**

家庭教育场景对信任极度敏感。若产品呈现测评感、打分感或诊断标签感，家长将抗拒深度输入。类咨询式、语音优先、不评判的采集方式，是获取高质量家庭上下文的前提。

以上假设共同构成育见的产品哲学与技术路线，后续所有功能设计均服务于验证这三项假设。

## 1.2 问题陈述：家长真正缺什么

从试点场景与内测反馈归纳，目标用户（焦虑但有反思能力的家长）面临五类结构性困境：

**第一，理解缺失。** 家长能看见行为（拖延、顶嘴、关门、沉默），却难以看见行为在家庭系统中的功能——例如，对抗是否是在争夺自主感，沉默是否是在避免冲突升级。

**第二，记忆断裂。** 每次与通用 AI 或咨询师对话，都需重新建立上下文。碎片观察无法自动串联为「这个孩子在这些情境下一贯如何反应」。

**第三，情绪与事实混杂。** 家长在高焦虑状态下，常把「懒」「不自觉」「不懂事」等评价当作孩子事实。若 AI 采信这些评价，将迅速失去信任并给出错误建议。

**第四，行动断裂。** 家长听懂道理，却常在当晚不知如何开口。缺少「今晚可试」的小步行动与沟通前预演，理解无法转化为行为改变。

**第五，越记录越乱。** 育儿笔记、聊天记录、零散咨询无法形成综合判断。家长需要的是机制化理解，而非更多信息堆叠。

育见针对上述困境，不是再提供一个「更会说话的 AI」，而是构建一个**长期理解这个孩子的系统**。

## 1.3 产品边界：我们刻意不做什么

为避免投资人与合作方误解，须明确边界：

育见**不做**心理测评与打分评级，不做医疗或心理咨询承诺，不替代学校教学与作业批改，不提供纯百科式育儿鸡汤。

育见**做**培优型家庭理解：服务「已经努力、想真正理解孩子」的家长；把真实生活片段转化为机制链与可验证行动；通过 SecondMe 长期沉淀，形成越用越懂的闭环。

这一边界既是伦理选择，也是商业定位——我们进入的是「家庭理解基础设施」，而非「危机干预」或「学科辅导」红海。

## 1.4 解决方案概述

育见通过以下路径验证核心假设：

1. **类咨询式采集**：四模块 Onboarding（日常节奏、学习作业、亲子沟通、家庭支持）+ 日常语音/文字片段输入，降低叙述门槛。
2. **结构化理解**：多 Agent 后台将叙述拆解为事实、情绪、评价、期待与原话，形成原子事实网络（FactAtom + EvidenceEpisode）。
3. **FamilyModel 构建**：综合证据网络、家庭互动循环、待验证假设与家长叙事模式，形成家庭认知模型。
4. **SecondMe 沉淀**：生成家长向深度摘要与画像，作为每轮交流的理解底座。
5. **可验证闭环**：交流 → 今晚可试任务 → 沟通预演 → 画像回顾 → 记忆更新，使理解可检验、可修正。

# 第二章 市场背景与验证机会

## 2.1 宏观背景：为什么现在是合适窗口

**政策与结构变化。** 双减实施后，学习支持与情绪劳动更多回归家庭；学校、家庭与社会对「家校共育」的工具需求上升，但学校不宜承担过重家庭教育干预，需要轻量、可信、非教学替代的数字支持。

**技术变化。** 大模型显著降低了自然语言理解与生成成本，使「家庭叙述 → 结构化理解」在经济上成为可能。但 C 端教育场景的瓶颈已从「能不能回答」转向「能不能长期理解这个孩子」——这正是育见切入点。

**用户变化。** 家长已习惯使用豆包、DeepSeek 等通用 AI，但普遍感受到「说得对，但不像我家」的落差。这为「家庭上下文 + 长期记忆」类产品创造了认知空间。

## 2.2 目标市场与用户画像

**主用户：** 焦虑但有反思能力的家长，通常为 K12 阶段孩子的父母。

**双重角色：** 他们既是日常管教者（催作业、管手机、处理冲突），也是观察者（想理解孩子为什么这样、家庭互动在哪里形成循环）。

**典型心理语句：** 「我已经很努力了，但我想真的理解孩子，不是再听一套大道理。」

**早期切入场景（已验证方向）：**

- 作业拖延与催促升级
- 手机使用冲突
- 沟通前焦虑（「今晚想谈，不知道怎么开口」）
- 支教期间家庭沟通变化（九州问学等公益场景）
- 社区/学校共育场景下的自愿体验（丰台试点）

## 2.3 市场规模与量化机会

【待团队补全：TAM / SAM / SOM 及权威数据来源】

本母稿不编造全国家庭教育市场规模数字。定性判断如下：

- **潜在用户基数大**：K12 家庭普遍面临亲子沟通与学业支持压力。
- **付费意愿待验证**：家长为「理解孩子」付费的逻辑成立，但价格带、频次与竞品替代需通过试点验证。
- **B2B2C 通道可行**：学校、支教组织、社区与教育机构可作为触达与背书渠道，降低 C 端获客成本——当前公益试点即在此验证。

## 2.4 传统解法为何不够

**通用大模型（豆包、DeepSeek、ChatGPT 等）**

优势在于泛化知识与流畅对话；劣势在于缺乏持久家庭模型、难以锚定本家庭原话与历史、容易输出泛化建议，且无法形成交流—任务—预演—画像的产品闭环。

**传统教育咨询**

优势在于专业性与深度关系；劣势在于频次低、成本高、难沉淀结构化记忆、难覆盖日常碎片场景，规模化受限。

**记录类育儿应用**

优势在于留存片段；劣势在于缺少 AI 深度建模、机制推理与可验证行动设计。

育见的位置是：**用产品化方式，提供接近咨询深度的家庭理解，并具备通用 AI 的可及性与可持续记忆。**

## 3.2 完整用户旅程（叙事版）

一名新家长进入育见后，经历以下阶段：

**阶段一：建立信任。** 在 intro 页了解产品价值——这不是测评，也不是说教，而是像「带过不少学生的师兄师姐」与家长面谈。柔和的视觉与分步设计降低防备心。

**阶段二：基础信息。** 填写孩子昵称、年级等基础语境，使后续理解有锚点。

**阶段三：四模块深采集。** 按顺序完成四个模块，每个模块包含「自由叙述 → AI 面谈式追问 → 阶段总结确认」：

1. *孩子平时怎么过* — 上学日/周末节奏、手机、睡眠、疲劳线索  
2. *学习和作业怎么进行* — 从「准备开始」到「最后收场」的真实过程  
3. *你们通常怎么沟通* — 典型对话原话、升级与冷场方式  
4. *家里怎么一起支持他* — 家长分工、规则落地、试过办法与效果  

> 说明：当前 hi-fi UI 为四模块。后端记忆层保留五入口历史命名（含情绪压力等），情绪维度已融入沟通与日常模块，属工程兼容而非家长侧重复填写。

**阶段四：综合建模。** 系统在 generating 页调用 synthesis 与 diagnosis 管线，将四模块证据综合为初始画像与机制候选，写入家庭记忆层。

**阶段五：解锁主站。** 家长进入四 Tab 长期使用：

- **交流（/daily）**：描述真实片段，获得锚定历史的流式理解  
- **任务（/tasks）**：沉淀「今晚可试」的小步行动  
- **预演（/rehearsal）**：模拟「准备说的话，孩子可能怎么听」  
- **画像（/family-profile）**：查看 SecondMe 摘要、机制链与回顾  

**阶段六：长期飞轮。** 每次交流触发检索、编排与流式生成；高价值信息异步写入 Episode/Atom 与记忆层；任务反馈与反证信号可修正既有假设。家长感知为「越聊越贴我家情况」。

## 3.3 日常交流：产品体验的关键设计

交流页是育见最高频场景，其设计直接决定用户是否相信「系统真的在懂我家孩子」。

**Thinking 四宫格。** 在首字出现前，系统展示正在进行的理解步骤（如读取历史、比对场景），避免「秒回鸡汤」的信任损耗。

**短 prose + 结构化 section。** 正文保持面谈语气、篇幅克制；随后流出结构化模块（如判断依据、今晚可试），满足反思型家长对「依据」的需求。

**高置信与低置信分轨。** 信息充分时，给出锚定画像与历史的分析；信息不足时，不硬答，而是给出可能方向并**每轮最多一个主追问**，先区分 A/B 再深入。

**Actions 与输入解锁。** 结构化内容落定后给出可行动建议，并解锁下一轮输入——家长不必等待所有后台任务完成。

**WarmTurn 同线程多轮。** 同一会话后续轮次复用检索缓存，使连续追问更连贯、成本更可控。

## 3.4 典型场景举例

**场景 A：作业拖延爆发。** 家长输入：「催他就发脾气，不催就不写。」系统引用 Onboarding 中的作业流程与沟通原话，拆解当下更可能属于「催促升级」还是「疲劳逃避」，并给出今晚可试的低冲突切入点。

**场景 B：沟通前焦虑。** 家长准备谈手机使用，不知从何说起。预演模块模拟孩子可能的反应方式，降低误判与冲突升级风险。

**场景 C：反证信号。** 家长提到「今天居然主动写了两题」。系统识别为 counter_evidence，修正「孩子完全不动」的过度概括，避免固化偏见。

## 3.5 产品价值总结

对家长而言，育见提供的不是更多信息，而是**三种稀缺能力**：

1. **机制化理解** — 看见行为在家庭系统中的功能  
2. **情境化记忆** — 把碎片串成属于这个孩子的叙事  
3. **可验证行动** — 把理解转化为今晚能做的一小步

# 第四章 技术体系：假设如何被工程化实现

## 4.1 技术定位

育见的技术本质不是「一个更强的 Chatbot Prompt」，而是**家庭场景理解基础设施**：由采集层、语义理解层、记忆层、编排层与反馈层构成的全栈系统。技术服务于三个目标——**降低幻觉、累积理解、可审计演进**。

## 4.2 总体架构

```
用户层（Web / 小程序）
    ↓
交互采集层（语音 ASR、Onboarding、日常输入）
    ↓
BFF 编排层（/api/daily/stream、安全分级、路由决策）
    ↓
语义理解层（entry_evidence、episodeExtractor、dailyDecompose 等）
    ↓
家庭事实层（EvidenceEpisode + FactAtom + 分层记忆）
    ↓
FamilyModel（证据网络、互动循环、假设、SecondMe 摘要）
    ↓
智能建议层（前台 LLM 锚定事实生成 + 任务/预演）
    ↓
反馈更新层（memory_write、episode_ingest、任务反馈）
```

## 4.3 FamilyModel：不是标签，是家庭认知模型

**FamilyModel** 是产品概念名，指持续积累家庭事实、理解家庭关系、记录孩子成长变化、支撑未来建议的**家庭认知模型**。

工程上，它分散存储于 PostgreSQL `memory_layer_items` 的多个 layer（如 `entry_evidence_packs`、`evidence_networks`、`family_interaction_cycles`、`pending_hypotheses`、`deep_model_digest` 等），**并非名为 FamilyModel 的单一数据表**。

其价值在于：同一个「拖延」，在不同家庭中对应不同的触发—反应—解读循环；只有建立家庭级模型，建议才能从人口统计意义上的正确，变为情境意义上的有效。

## 4.4 原子事实网络

系统将家长叙述拆解为两类核心单元：

**EvidenceEpisode（证据片段）**：语义完整的场景单元，是向量检索的主粒度，保留「当时在发生什么」的语境。

**FactAtom（原子事实）**：最小可追溯陈述，标注来源类型（家长明确陈述、家长推测、孩子原话、材料观察等）、事实类型与证据强度。

对比说明：

- 普通记录：「孩子学习不好。」  
- 原子网络：数学近期失分点 + 连续三日作业时长 + 催促次数变化 + 孩子抵触原话 + 一次主动完成的反证  

高价值 Atom（如孩子原话、反证、执行反馈）单独向量化，支撑精细检索与机制验证。

## 4.5 语义理解：区分事实、情绪、评价与期待

家庭叙述天然混杂多层信息。育见在 Agent 层明确拆分：

| 信息类型 | 示例 | 处理原则 |
|----------|------|----------|
| 事实 | 「最近三天写到 11 点」 | 可验证事实，进入推理链 |
| 情绪 | 「我真的很焦虑」 | 记录家长状态，**不**等同孩子状态 |
| 评价 | 「他就是不努力」 | 记为家长解释，**不**采信为孩子事实 |
| 期待 | 「希望他主动学习」 | 记为目标，用于理解家庭压力来源 |
| 孩子原话 | 「知道了别说了」 | 高价值锚点，优先检索 |

这一原则是信任与准确性的底线，亦写入 Prompt 宪法（`parentFacingStyle`）与后台抽取契约。

## 4.6 长期记忆与检索

记忆体系包括：

- **分层 JSON 记忆**（`memory_layer_items`）：画像、机制矩阵、互动循环、假设等  
- **向量表**（`evidence_episodes` / `fact_atoms` + pgvector）：语义召回，可降级  
- **回合快照**（`turn_events`）：支持审计与回放  
- **异步任务队列**（`job_queue`）：memory_write、episode_ingest 可靠重试  

检索策略采用三级降级：Episode 向量检索 → 精排 → 最近对话兜底；同线程 warmTurn 复用缓存以优化体验与成本。

## 4.7 推理机制：规则编排 + 检索 + LLM

育见**不是**端到端黑盒模型一键输出结论，而是：

1. **规则编排层**（无 LLM）：安全分级、输入分类、与已有模型关系判断、路由决策（是否追问、响应类型）  
2. **检索打包**：每轮前台仅读取 retrievalPack + deepModelDigest，防止越权编造  
3. **后台 LLM**：synthesis、diagnosis、deep_mechanism_review 更新机制矩阵与假设  
4. **前台 LLM**：在门控下流式生成，强制引用家庭事实（`read-contract` 契约）  

低置信时不硬答的设计，亦在编排层实现，而非仅靠 Prompt 约束。

## 4.8 技术栈与工程成熟度

| 类别 | 当前实现 |
|------|----------|
| 前端 | Next.js、TypeScript、hi-fi 设计系统 |
| 后端 | Node.js、server-only 模块、BFF 流式编排 |

# 第七章 竞争优势与壁垒

## 7.1 三方对比

| 维度 | 通用大模型 | 传统教育咨询 | 育见 |
|------|------------|--------------|------|
| 核心能力 | 泛化问答 | 人工深度会谈 | 长期家庭建模 |
| 记忆 | 会话级/有限 | 咨询师个人记忆 | 分层记忆 + SecondMe |
| 上下文 | 通用知识 | 单次会谈经验 | 本家庭事实与机制链 |
| 输出形态 | 长回答、建议堆砌 | 专业但低频 | 短面谈 + 结构化依据 + 可试行动 |
| 输入门槛 | 低（但浅） | 高（预约、费用） | 语音叙述真实片段 |
| 可验证性 | 弱 | 中 | 强（任务、预演、反证修正） |
| 规模化 | 高 | 低 | 产品化路径清晰 |

## 7.2 六项可辩护优势

1. **从 RAG 到 FamilyModel** — 不止检索相似文本，而是构建机制链与互动循环。  
2. **多 Agent 分工 + 前台门控** — 43 个专用 Prompt，前后台分离，降低幻觉。  
3. **原子事实网络** — 非结构化家庭叙述可计算、可修正（counter_evidence）。  
4. **类咨询式 Onboarding** — 四模块面谈采集，建立「懂我家」信任基础。  
5. **交流—任务—预演—画像闭环** — 理解可留存、可行动、可回顾。  
6. **公益试点数据飞轮** — 真实支教与社区场景驱动迭代，非纯实验室产品。  

## 7.3 壁垒的诚实评估

**已有壁垒（早期）：**

- 家庭教育领域 Prompt 与记忆契约体系（迁移成本高）  
- 分层记忆 + Episode/Atom 工程实现  
- 公益场景渠道与清华 iCenter 背书  
- 产品体验气质（不评判、面谈式）形成的用户信任  

**尚未形成壁垒（需持续建设）：**

- 用户规模与网络效应  
- 专有数据规模（当前内测规模仍小）  
- 品牌认知与付费转化  
- 监管与合规体系的产品化  

## 7.4 安全、合规与伦理

| 领域 | 策略 |
|------|------|
| 风险输入 | safety tier 分级；critical 走安全响应路径 |
| 标签化 | 不采信家长评价词为孩子事实 |
| 诊断边界 | 不做心理诊断标签；机制用生活语言 |
| 数据隔离 | 家庭级租户隔离；对外仅匿名聚合 |
| AI 标识 | 界面标注 AI 生成，仅供参考 |
