## 2. 逐项功能测评（对应优化清单编号）

### P0-1 前台输出收敛为 visibleReply + traceId
- **目标**：`/api/daily` 不泄露 `decomposedInput`/`routingDecision`/`memoryAction`/置信度等内部字段。
- **步骤**：POST `/api/daily`（带登录态），观察返回 JSON 顶层字段。
- **预期**：仅 `{ ok, data: { traceId, visibleReply, linkedAreas } }`；无任何内部推理字段、无 confidence。
- **状态**：✅（build 验证 + 字段对齐 observation 页 `visibleReply`）

### P0-2 后台记忆写入异步化
- **目标**：用户看到回复不被记忆写入阻塞。
- **步骤**：发一条 daily，记录首字节/响应到达时刻；后台写入失败时观察前台是否受影响。
- **预期**：回复先到，写入在后台；写入失败仅日志，不影响前台返回。
- **状态**：✅

### P0-3 日常对话前台流式输出
- **目标**：`/api/daily/stream` NDJSON 流式；无 key 降级为规则文案不退化；风险输入走安全文案。
- **步骤**：①档位 B 发一句普通家庭困扰 → 看是否逐行流式；②档位 A（无 key）发同样输入 → 看是否规则文案且不空；③发自伤/危机类输入 → 看是否安全文案。
- **预期**：①真流式逐行 `final`；②降级文案完整可用；③安全文案优先。
- **状态**：✅（A 档降级路径已验；B 档真流式需带 key 环境复测）⚠️ 真流式

### 功能-1 家庭综合规划 family_planner
- **目标**：输出 1-3 个可承受动作；先稳边界；失败原因不清先追问（文档 5.4.3）。
- **步骤**：POST `/api/family-planner`：
  - 清晰失败节点输入：「孩子作业拖到半夜，之前定了晚饭后开始但我加班没人盯就崩了」
  - 模糊输入：「孩子学习不行，帮我规划下」
- **预期**：清晰输入 → `enoughToPlan=true` + 1-3 个 actions + `boundaryFirst` 非空；模糊输入 → `enoughToPlan=false` + `actions=[]` + `missingInfo` 追问失败节点。
- **状态**：✅（docker 实测：清晰→2 动作+边界；模糊→0 动作+追问）

### 功能-2 家庭支持看板 board
- **目标**：五段式看板由后台生成、读持久化快照，不前端伪造；信息不足温和空态。
- **步骤**：档位 C 走完几轮 daily 后 GET `/api/board`（带登录态）。
- **预期**：返回当前状态/稳定理解/互动模式/待验证点/下一步五段 + `version`/`evidenceRefs`/`updatedAt`；空数据时不硬编画像。
- **状态**：✅（见闭环-1 docker 验证）

### 架构-1 三层语义检索（EvidenceEpisode + FactAtom + pgvector）
- **目标**：语义完整 Episode 召回（非碎片）；FactAtom 标 sourceType；选择性向量化；跨用词语义召回。
- **步骤**：
  - 档位 B：`EMBEDDING_API_KEY=... node scripts/test-vector-retrieval.mjs`（向量精排 vs 取最近）
  - 档位 C：`DATABASE_URL=... EMBEDDING_API_KEY=... node scripts/verify-pgvector.mjs ["逃避"]`
  - psql 核查：`evidence_episodes` 有行带 embedding；`fact_atoms` 标 source_type；高价值 Atom embedding 非 NULL、普通 Atom embedding=NULL。
- **预期**：向量精排 top5 平均相关度显著高于取最近（实测 0.61 vs 0.48，+27%）；query「逃避」召回「背不会/摔本子」场景（跨用词）。
- **状态**：✅（docker 端到端通过）

### 架构-2 Prompt 外置（md 源 + 构建期生成）
- **目标**：11 个 prompt 由 `prompts/{front,background}/*.md` 编译进 bundle，零运行时 fs。
- **步骤**：改任一 `prompts/**/*.md` → `npm run prompts` → 看 `registry.generated.ts` 是否重生成；grep 生成文件无乱码（U+FFFD）。
- **预期**：11 个 prompt 全量生成，关键硬规则（dailyDialogue 的 Q2 追问、episodeExtractor 来源底线、familyPlanner 1-3 动作）保留。
- **状态**：✅

### 架构-3 traceId 全链路字段闭环
- **目标**：一条家长消息的 traceId 贯穿 job_queue / daily_updates / evidence_episodes。
- **步骤**：档位 C 发一条 daily 拿到 `traceId`，psql：
  ```sql
  SELECT job_type, trace_id FROM job_queue WHERE trace_id='<traceId>';
  SELECT data->>'sourceEventId' FROM memory_layer_items WHERE layer_name='daily_updates';
  SELECT source_event_id FROM evidence_episodes ORDER BY created_at DESC LIMIT 1;
  ```
- **预期**：两个 job 共享同一 trace_id；daily_update.sourceEventId == traceId；episode.source_event_id == 同一 traceId。
- **状态**：✅（docker 三方对齐）
