## 一、项目定位（30 秒版 · 3 句话）

1. **育见**是面向下一代家庭教育场景的智能理解系统，帮助家长从「管教焦虑」走向「真正理解这个孩子」。
2. 通过类咨询式对话获取完整家庭场景上下文，经语义解析与多 Agent 深度建模，构建可长期演进的 **FamilyModel** 与孩子专属 **SecondMe**。
3. 实现 AI 从「简单回答问题」到「长期理解孩子、辅助家庭决策」的转变——不是育儿鸡汤，而是锚定家庭事实的机制化理解。

### 一句话介绍（BP 封面）

> 育见：基于家庭场景深度建模的下一代 AI 家庭教育理解系统，为每个孩子构建可计算的 SecondMe，长期辅助家庭成长决策。

### 团队对外技术表述（可改写进投资亮点）

我们团队开发了一款面向下一代家庭教育场景的智能理解系统「育见」。通过**类咨询方式**获取完整的家庭场景上下文，利用**语义解析、权重系数与向量检索**技术，将非结构化输入转化为**可计算推理的原子事实网络（Fact Atom Network）**，构建深度感知的 **FamilyModel**，打造孩子专属教育 **SecondMe**，实现 AI 从「简单回答问题」向「长期理解孩子、辅助家庭决策」的转变。

## 二、问题与机会（需求痛点 · 供 BP 第四章引出第五章）

### 2.1 家长侧痛点

| 痛点 | 具体表现 | 现有解法为何不够 |
|------|----------|------------------|
| **理解缺失** | 看见拖延、发脾气、沉默，不知道背后机制 | 通用 AI 给「多鼓励、定计划」，不贴合这个孩子 |
| **记忆断裂** | 每次咨询/聊天从零开始 | ChatGPT/豆包不持久记住家庭结构与历史片段 |
| **情绪与事实混杂** | 家长焦虑、贴标签（懒、不自觉）被当成事实 | 缺少拆解「家长情绪 vs 孩子行为 vs 家长推测」的能力 |
| **行动难落地** | 听懂道理，今晚仍不知怎么开口 | 缺少「今晚可试」的小步验证与预演 |
| **越用越不懂** | 碎片记录多，综合判断少 | 没有越积越准的孩子模型 |

### 2.2 行业/场景机会

- **K12 家庭教育**：双减后家庭承担更多学习支持与情绪劳动；青春期亲子冲突高发。
- **AI 教育**：大模型普及，但 C 端教育场景需要从「问答」升级到「长期陪伴式理解」。
- **公益与学校共育**：支教、家校沟通需要可信、轻量、可沉淀的数字工具（当前公益试点：九州问学 30+ 支队、北京丰台区学校/社区）。

### 2.3 产品边界（避免投资人误解）

**育见不做：**

- 心理测评、打分评级、诊断贴标签
- 替代学校教学、作业批改
- 医疗/心理咨询承诺
- 纯鸡汤式育儿百科

**育见做：**

- **培优型**家庭理解：服务「已经努力、想真正理解孩子」的家长
- 把真实生活片段 → 机制链 → 可验证小步行动
- 长期 SecondMe 沉淀，越用越懂

## 三、目标用户与核心任务

### 3.1 用户画像

**主用户**：焦虑但有反思能力的家长（通常为父母）。

**双重角色**：

1. **日常管教者**：催作业、管手机、处理学习冲突
2. **观察者**：想理解孩子为什么这样、家庭互动哪里在循环

**典型心理**：「我已经很努力了，但我想真的理解孩子，不是再听一套大道理。」

### 3.2 用户核心任务（优先级）

1. 获得比通用 AI **更深的孩子理解**（拆开表面行为，看见家庭机制链）
2. **促进亲子关系**（从对抗到看见，从管教到理解）
3. 帮助孩子提高成绩（产品方向存在，**当前版本不直接主打**，避免夸大）

### 3.3 典型使用场景（可画进 BP 用户旅程图）

| 场景 | 家长行为 | 育见响应 |
|------|----------|----------|
| 作业拖延爆发 | 「催他就发脾气，不催就不写」 | 拆解现场、区分 A/B、给锚定历史的理解 |
| 沟通前焦虑 | 「今晚想跟他谈手机，不知道怎么开口」 | 预演：孩子会怎么听 |
| 碎片观察 | 语音记录今天发生的一件事 | 写入记忆层，更新 SecondMe |
| 长期困惑 | 「是不是我太焦虑了？」 | 接住情绪但不当孩子事实，追问可验证现场 |
| 反证信号 | 「他今天居然主动写了两题」 | 识别 counter_evidence，更新假设而非固化偏见 |

## 四、完整用户使用逻辑（极详细 · BP 商业模式核心素材）

### 4.1 产品形态与入口

| 端 | 状态 | 说明 |
|----|------|------|
| **Web** | 已上线 | https://yujian.yihe.site ，hi-fi 四 Tab 主站 |
| **微信小程序** | 对齐中 | Taro 3 + React，复用同一 BFF API |
| **登录** | 手机号/微信 | Web Cookie；小程序 Bearer Token |

### 4.2 新用户 Onboarding（五入口画像 · 必做门禁）

**目的**：在首次深度交流前，用类咨询面谈（非冷冰冰问卷）建立初始 FamilyModel。

**路径**：

```
登录 / 注册
  → /profile/build/intro（建立信任、说明价值）
  → /profile/build/basic（孩子基础信息）
  → /profile/build（五入口 Hub）
  → 分模块采集（每模块：输入 → AI 追问 → 阶段总结）
  → /profile/build/final-follow-up（最终确认）
  → /profile/generating（调用 synthesis + diagnosis 生成画像）
  → /profile/result（展示初始 SecondMe）
  → 解锁四 Tab 主站
```

**五入口模块**（覆盖家庭场景全貌）：

| 入口 | 主题 | 采集内容方向 |
|------|------|--------------|
| 学习作业 | homework | 学习节奏、作业冲突、催促与对抗 |
| 日常节奏 | daily | 作息、手机、生活节奏 |
| 亲子沟通 | communication | 说话方式、冲突模式、孩子反应 |
| 情绪压力 | emotional | 压力来源、情绪表达、崩溃点 |
| 关系环境 | family | 家庭结构、支持分工、环境因素 |

**采集设计原则**：

- **语音优先**，短输入友好；不必组织成「专业描述」
- 每模块软引导 **≥800 字**（面谈式追问，非硬阻断问卷）
- 保留家长与孩子**原话**，作为后续推理锚点

**Onboarding 心理路径（BP 可强调）**：

1. **建立信任**：柔和视觉、非测评语气、清北学长/师姐面谈气质
2. **降低压力**：分模块、可暂停、像聊天不像填表
3. **引导投入**：五入口保证上下文完整，避免「AI 不了解我家就瞎建议」

### 4.3 主站四 Tab 使用逻辑

#### Tab 1：交流（/daily）—— 最高频核心

**家长操作**：

1. 打开交流页，看到空态引导或历史线程
2. 按住说话或切换文字，描述**真实片段**（如「昨天数学作业又拖到 11 点」）
3. 等待系统响应（先 thinking 四宫格，再流式正文）
4. 阅读：短回复（prose）+ 结构化 section（判断依据/画像分析/追问等）
5. 查看 actions（「今晚可试」、深度展开入口）
6. 可继续追问，形成 **warmTurn** 同线程多轮（系统复用检索缓存，越聊越贴家庭模型）

**系统后台（用户不可见）**：

```
家长输入
  → 编排管道 runOrchestrationPipeline（安全检测、输入分类、记忆检索、关系判断、路由）
  → 并行：digest 加载/构建 + 编排
  → 单次 LLM 流式：prose + visible sections（合并调用，体验无缝）
  → hidden sections 后台预取（深度展开点开即读）
  → actions 生成
  → 选择性记忆写入（非每轮都写，省成本）
  → turn_event 快照落库
```

**单轮流式事件顺序（产品体验关键）**：

```
start → thinking（四宫格 chips）
  → delta（正文流式）
  → prose_complete
  → section_start / section_delta / section_complete（≤3 个可见 section）
  → sections_complete
  → actions
  → final
```

**交流页设计意图**：

- thinking 先出现：让家长感知「系统在读你家孩子」，不是秒回鸡汤
- prose 短、有人味：像师兄师姐面谈，不是报告体
- section 结构化：满足「想看清判断依据」的反思型家长
- actions 后解锁输入：保证家长先看完本轮关键内容

#### Tab 2：任务（/tasks）

**来源**：交流中沉淀的「今晚可试」任务。

**家长操作**：

- 查看待试任务列表（状态、来源）
- 完成任务后提交反馈（是否尝试、孩子反应、家长感受）
- 反馈回流记忆系统，用于验证/修正假设

**价值**：把「理解」变成**可验证**的小步行动，形成闭环。

#### Tab 3：预演（/rehearsal）

**场景**：家长准备对孩子说一段重要的话（谈手机、谈成绩、谈规则）。

**家长操作**：

1. 选择预演场景
2. 输入准备说的话
3. 系统分析「孩子可能会怎么听」——不是帮孩子回嘴，而是帮家长看见**接收视角**
4. 可选：给出「您可以这样说」的改写建议

**价值**：降低沟通前焦虑，减少误判与二次冲突。

#### Tab 4：画像（/family-profile）

**内容**：长期 SecondMe 的家长可见层。

- 摘要卡、机制链、家庭互动循环
- 本周回顾、结构张力（家庭运转张力，人话呈现）
- 深度展开：hidden section 预取内容、证据与假设

**价值**：让家长看见「越来越懂孩子」的轨迹，而非每次对话孤立。


## 七、核心技术模型（详细 · BP 技术壁垒核心）

### 7.1 概念定义

| 概念 | 定义 | 家长可感知形态 |
|------|------|----------------|
| **FamilyModel** | 家庭层面的结构模型：互动循环、支持分工、叙事模式、机制假设 | 画像中的机制链、家庭模式描述 |
| **SecondMe** | 孩子专属的深度理解档案，随输入持续演进 | 画像摘要、交流中的「越来越懂你」 |
| **原子事实网络** | 从非结构化输入拆解的可计算事实单元（FactAtom）及语义完整 Episode | 后台推理；前台以「你之前提到过…」体现 |
| **deepModelDigest** | 家长向 SecondMe 摘要层，多 Agent 共建 | 交流/画像的思考 chips 与锚定引用 |
| **retrievalPack** | 每轮交流检索打包，前台 AI 唯一事实来源 | AI 回复中具体场景与原话 |
| **机制链** | 家长动作 → 孩子反应 → 可能在保护什么 | section 中的结构化分析 |

### 7.2 非结构化输入 → 可计算推理（技术路径）

**Step 1：采集与拆解**

- 家长自然语言 / 语音转写进入系统
- `entry_evidence` Agent：拆解为 `verifiableFacts`（可验证事实）、`childBehaviors`（孩子行为）、`triggerPoints`（触发点）
- 日常交流：`episodeExtractor` 抽取语义完整 **EvidenceEpisode** + 高价值 **FactAtom**
- 原话片段单独存储，强制在 digest 中可召回

**Step 2：证据网络与机制矩阵**

- `profileBuildSynthesis`：四模块证据合成 **candidateMechanismMatrix**（候选机制矩阵）
- `profileBuildDiagnosis`：生成 coreJudgment、条件画像、家庭互动循环草案
- `deep_mechanism_review`：读全量记忆，覆盖/深化机制层，更新假设与家长叙事模式

**Step 3：权重与检索（语义解析 + 权重系数）**

- **向量语义检索**：Episode 为主召回单元（语义完整，非碎片）；FactAtom 高价值单独向量化
- **三段式精排**：标签过滤 → 向量 topK → 证据/时间加权
- **降级链**：pgvector → 应用层 cosine → 取最近（保证无 embedding 时仍可用）
- **warmTurn 缓存**：同线程后续轮复用检索 packet，增量合并新输入，避免丢掉画像与向量证据

**Step 4：编排路由（非 LLM 黑盒）**

`runOrchestrationPipeline` 在调用前台 LLM 前完成：

- 安全分级（critical / elevated / relief）
- 输入类型分类（daily_observation / ask_advice / counter_evidence / emotional_vent 等）
- 与已有模型关系判断（insufficient / old_mechanism_repetition / counter_evidence / new_mechanism_signal 等）
- 路由决策：是否追问、响应类型、记忆写入策略
- **规则 + 检索先行，LLM 后行**：降低幻觉与误判

**Step 5：前台生成（锚定事实门控）**

前台 parentFacing Agent 强制：

1. 先读 `deepModelDigest` + `retrievalPack`
2. 至少引用 1 条 anchoredFacts / entryFacts
3. 至少 1 句机制闭环（人话，非诊断标签）
4. 培优语气；禁止空泛鸡汤
5. 低置信每轮最多一个主追问，先说明「想区分 A 还是 B」

### 7.3 多 Agent 分工表（完整）

| Agent | 触发时机 | 输入 | 输出写入 |
|-------|----------|------|----------|
| 四模块采集 Agent | Onboarding 每模块 | 模块 SP + 家长输入 | entry_records、stage summary |
| entry_evidence | 每模块完成 | rawText | 结构化事实包 entry_evidence_packs |
| profileBuildSynthesis | 四模块齐 | 四份 stage summary | 证据网络草案、机制矩阵草案 |
| profileBuildDiagnosis | synthesis 后 | 交接物 | coreJudgment、条件画像、互动循环草案 |
| deep_mechanism_review | 建模完成 + 周期性 | 全量记忆层 | 覆盖机制矩阵、假设、家长叙事 |
| deepModelDigestBuilder | deep 完成 + 日刷新 | 机制+画像+循环 | deep_model_digest 家长向摘要 |
| dailyPortraitRefresh | 进 daily/profile | digest + 近期输入 | portraitCards、thinkingChips |
| episodeExtractor | 高价值日常轮 | 家长输入文本 | EvidenceEpisode + FactAtom |

### 7.5 Prompt Cache 与性能工程（体现技术成熟度）

- 稳定 system prompt 前缀 + 动态 user payload 后缀，利于 **DeepSeek 自动 prefix cache**
- 线上实测：命中率 p50 ~84%，~4k tokens 复用，降低长上下文成本
- 流式 NDJSON：`/api/daily/stream`，prose + section **单次 LLM 调用** marker 流，减少排队
- 首字延迟主因是 **模型 prefill**，非 BFF 扣留（`providerToProseDeltaMs=0ms` 实测）

## 八、产品技术优势（投资亮点素材 · 建议 4–6 条）

### 亮点 1：从 RAG 到 FamilyModel——不是检索片段，是建模孩子

通用 AI：检索相似文本 → 生成建议。  
育见：检索 + 机制矩阵 + 互动循环 + 假设验证 → **SecondMe 持续演进**。

### 亮点 2：多 Agent 分工 + 前台门控——降低幻觉与「AI 味」

40+ 专用 Prompt；前台强制引用家庭事实；后台周期性 mechanism review；规则编排先于 LLM。

### 亮点 3：原子事实网络 + 语义 Episode——非结构化真正可计算

Episode（语义完整场景）+ FactAtom（高价值原子事实）+ 向量精排；支持 counter_evidence 修正偏见。

### 亮点 4：类咨询式 Onboarding——五入口建立可信上下文

非问卷式；面谈追问；软引导深度；解决「AI 不懂我家」信任问题。

### 亮点 5：交流—任务—预演—画像闭环——可验证、可留存

不是聊完即走；任务反馈印证假设；预演降低冲突；画像看见长期变化。

### 亮点 6：双端 + 公益试点验证——真实场景数据飞轮

Web 已上线；小程序对齐中；清华 iCenter 公益试点；支教 30+ 支队 + 丰台学校/社区。
