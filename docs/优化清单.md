## 第一批：P0 红线（已完成）

### ✅ P0-1 前台输出收敛为 visibleReply + traceId 🔴
- **问题**：`app/api/daily/route.ts` 原样返回整个 `orchestration` 对象，内含 `decomposedInput`、`routingDecision`、`memoryAction`、`relationshipToExistingModel`（含 confidence）等内部判断字段。
- **违反**：交付文档 6.5（前台只输出 `traceId + visibleReply`）、11.2（禁止暴露内部术语/置信度）、13.3 P0（前台输出复杂 JSON）。
- **修复**：route 收敛为 `{ ok, data: { traceId, visibleReply, linkedAreas } }`。`linkedAreas` 为家长可见的关联领域名（衍生展示，非内部推理），内部字段全部留在服务端。
- **顺带修复既有 bug**：前端 `app/observation/page.tsx` 原读 `orchestration.frontResponse`，而后端字段名是 `frontResponseDraft`——**字段名不匹配导致 AI 回复一直拿不到、永远 fallback 到 mock**。现已对齐为 `visibleReply`。
- **验证**：`npm run typecheck` 通过、`npm run build` 通过。

### ✅ P0-2 后台记忆写入异步化 🔴
- **问题**：`app/api/daily/route.ts:40` 在返回响应前 `await runMemoryWritePipeline()`，用户必须等记忆写入完成才看到回复。
- **违反**：交付文档 6.3（后台任务不得阻塞用户看到回复）、12.4、14.3。
- **修复**：改为 fire-and-forget（`void ...catch()`），先返回回复，记忆写入在后台执行；写入失败只记日志、不影响前台。当前部署为 pm2 fork 长驻进程，该方式安全可靠。
- **遗留**：失败重试机制留待 job_queue 改进（见 架构-4）。

## 第二批：核心链路（✅ 已完成）

### ✅ P0-3 日常对话前台流式输出 🔴（已完成 · commit 09bf368）
- **关键现状**：daily 此前是**纯规则引擎**（`orchestration/pipeline.ts` 未接 LLM），回复是规则模板。
- **实现**：新增 `app/api/daily/stream/route.ts`（NDJSON 流式）——规则引擎做检索/判断/安全检测/降级文案，LLM（`dailyDialogueOrchestration`）流式生成 visibleReply；**无 AI key 或失败时降级为规则文案，行为不退化**；安全风险输入直接走规则安全文案。前端 `observation` 改为逐行消费流、实时展示。后台写入异步。
- **注意**：真流式 LLM 效果需在配置了 `FAST_AI_API_KEY` 的环境验证；当前无 key 环境走规则降级路径，已通过 typecheck + build。旧 `/api/daily`（非流式）保留未删。

### ✅ 功能-1 家庭综合规划 family_planner 🟡（已完成 · commit 5fd151a）
- **实现**：新增 `app/api/family-planner/route.ts`（`familyPlanner` prompt + LLM + 降级文案，actions 强制收敛 1-3 个）+ `app/family-planner/page.tsx`（场景化自述工作台：大输入框 + 提示卡，符合 5.4.1）+ home 底部导航入口。输出严守 5.4.3「不给密集时间表、先稳边界、失败原因不清先追问」。前台只返回 `plan + traceId`。

### ✅ 功能-2 家庭支持看板 board 🟡（已完成 · commit 9d03d81）
- **实现**：新增 `app/api/board/route.ts`（`boardUpdater` prompt，五段式：当前状态/稳定理解/互动模式/待验证点/下一步 + 温和空态降级）+ `app/board/page.tsx`（五段卡片，数据全部来自后端）+ home 顶部入口卡。看板由后台生成、不由前端伪造；信息不足时不硬编画像。独立于现有 `family-profile` 档案页，零侵入。

## 第三批：架构深化

### ✅ 架构-1 向量检索（检索时精排版）🟠（已完成 · 实测提升 27%）
- **已做**：新增 `src/lib/server/memory/embedding.ts`（阿里百炼 `text-embedding-v3` 客户端，1024 维，自动分批 ≤10，`cosineSimilarity` + `rankByRelevance`）。`buildDailyDialogueRetrievalPacket(query)` 从「取最近 5 条」升级为「从全部历史事实按语义相关性精排」；`orchestration/pipeline` 传入 `userText` 作为 query；embedding 未配置时自动降级为「取最近」，功能不受影响。
- **实测**：用真实 key 跑 `scripts/test-vector-retrieval.mjs`，向量精排 top5 平均相关度 **0.61 vs 取最近 0.48（+27%）**；并通过实测发现并修复了「批量上限 10 条」的真实 bug。
- **遗留（后续可选）**：当前为检索时**实时编码**（无持久化），数据量大时应改为写入时预编码 + pgvector 索引（维度 1024 适配 pgvector），避免重复编码。
- **✅ 已升级（commit 6dde9e3）**：实时编码已被「**EvidenceEpisode + FactAtom + pgvector 三层语义检索**」取代——每轮输入后台 LLM 抽取语义完整 Episode（总向量化，主召回单元）+ FactAtom（标 `source_type`，仅高价值单独向量化）；检索三段式（标签过滤 + 向量 topK + 证据/时间加权精排）；三级降级链（pgvector→应用层 cosine→取最近）。新增两层为检索专用，现有 10 层与画像生成链路不动。docker pgvector 端到端验证通过：Episode 语义完整召回（非碎片）、来源分层正确、选择性向量化生效、跨用词语义召回（query「逃避」召回「背不会/摔本子」场景）。

### ☐ 架构-2 Prompt 外置为 md 文件 🟠
- **问题**：Prompt 全硬编码在 `src/lib/server/agent-prompts.ts` 等 .ts 文件。
- **缺口**：交付文档 8.2 要求拆到 `/prompts/core`、`/prompts/front`、`/prompts/background`、`/prompts/knowledge`、`/evals`。
- **建议**：抽到 md 文件，便于非工程迭代与版本管理。

### ☐ 架构-3 统一 FamilyModel / TurnEvent 命名 🟠
- **现状**：用 10 层 memory + `ChildStructureModel`(L5)、`DailyInteractionUpdate`(L9) 替代，**实际比文档更细**，但命名与文档 TurnEvent / FamilyModel 不一致。
- **建议**：**不必硬套文档命名**。但应补一层统一 `traceId` 贯穿（前台→写入→检索），便于全链路追溯（文档 7.1 字段闭环）。当前 traceId 已在 P0-1 引入前台，需向下游 memory write 透传。

### ☐ 架构-4 后台任务队列与重试 🟠
- **问题**：无 job_queue，写入同步 fire-and-forget，失败不可重试。
- **缺口**：交付文档 14.3 要求 job_queue 有 pending/running/succeeded/failed/retrying 状态、失败可重试。
- **建议**：引入轻量队列（可复用 PG 表 + 轮询，避免引入 Redis 依赖），承接 P0-2 的失败重试。

## 第四批：工程质量隐患

### ☐ 质量-1 Demo ID 硬编码 🟢
- **问题**：`memory/database-manager.ts:29-30` 写死 `DEFAULT_FAMILY_ID='family_demo'`、`DEFAULT_CHILD_ID='child_demo'`，所有数据归一。
- **违反**：交付文档 14.2「家庭数据必须按 familyId/childId 隔离，不能跨家庭检索」。
- **建议**：从会话/鉴权上下文注入真实 familyId/childId，贯穿写入与检索。

### ✅ 质量-2 内存降级显式告警 🟢（已完成）
- **已做**：`database-manager.ts` 的 `getServerMemoryStore()` 在首次降级到进程内存时显式 `console.warn` 一次（交付文档 10.4），提示「数据未持久化、重启丢失」，不再静默。
- **遗留**：DB 写入失败本身由上层 `try/catch` 抛出并记录（非静默落内存），符合预期；「写库失败即硬报错」开关可作为后续可选项。

### ☐ 质量-3 过时测试脚本 🟢
- **问题**：`scripts/test-xiaoyin-*.mjs` 调用的 `/n` 端点在 app 路由中不存在（真实端点为 `/api/daily`）。
- **建议**：更新或归档过时脚本，避免误导。

### ✅ 质量-4 diagnosis / synthesis 写入异步化 🟠（已完成 · 含误判修正）
- **核查修正**：之前判断「diagnosis/synthesis 返回完整内部对象违反 6.5」**有误**。核查消费方 `app/profile/generating/page.tsx` 后确认：这两个 route **不是前台对话 route，而是画像生成管线的数据步骤**——前端深度消费其完整字段（`candidateMechanismMatrix`、`primaryMechanismChain`、`secondMeConditionalProfile` 等）在客户端组装孩子画像。其完整输出**正确且必要**，不应收敛。
- **已做**：将 `await runMemoryWritePipeline` 改为异步（交付文档 6.3），并去除前端从未使用的 `memoryWrite` 响应字段。画像生成返回更快、写入不阻塞，且统一了全部 route 的写入模式。
- **遗留观察**：画像在前端用内部机制字段组装（`generating/page.tsx`）属现有架构特征；若要彻底做到「前台不接触内部判断」，需把组装逻辑移到服务端，成本较大，列入长期项。
