# 语音失效复盘与传承经验（2026-07-14 ～ 07-15）

> **适用范围**：育见小程序「实时按住说话 / 交流语音」与「沟通预演 · 亲子整段录音上传」  
> **性质**：两次排障的完整事后复盘 + 可传承的排障方法论  
> **读者**：后续 Cursor / Trae / Codex / 人类开发者  
> **配套规则**：`.cursor/rules/voice-debug-code-first.mdc`、`.cursor/rules/voice-input-locked.mdc`

本文记录的不是「最终修好了什么」 alone，而是：**错了哪些判断、为什么错、用户纠正后如何（错误地）应对、哪些试错浪费了时间、以后必须怎么做。**

## 0. 先分清两条完全不同的链路

| 维度 | 实时对话（按住说话） | 亲子录音文件 |
|------|----------------------|--------------|
| 产品入口 | 交流 Tab / 构建 / 预演按住说话 | 沟通预演 → 亲子对话录音页 |
| 客户端核心 | `useTencentAsrInput.ts` | `useTapFileRecorder.ts` + `dialogue/index.tsx` |
| 录音形态 | 实时 PCM 帧推流 | 整段 mp3 临时文件 |
| 识别服务 | 讯飞实时 ASR（WSS） | 腾讯云文件 ASR（`uploadFile` → BFF `dialogue-transcribe`） |
| 传输 | `wx.connectSocket` → `wss://…` | `Taro.uploadFile` → `https://yujian.yihe.site/...` |
| 共享依赖 | 全局 `RecorderManager` + `recorderState` claim | 同左 |
| 失败时 UI | 「语音连接超时」等 | 「上传失败，请检查网络后重试。」等 |

**铁律**：两条链路可以**同时挂**，也可以**一个挂一个好**；排障时必须分开写假设，禁止「语音都挂了所以一定是同一个外部原因」。

## 1.1 最强旁证（2026-07-15 用户补充，状态腐烂模型）

> **2026-07-15 下午用户补充**：前天晚上微信平台正式版、昨晚最终稳定体验版语音均正常；但开发版若**几小时前扫码、真机调试早已断开**，实时语音与亲子文件均失败（语音连接超时）；若**立刻重新扫码**则正常。

这条线索的重要性：**它几乎直接证明「坏的是微信 进程内长期存活的开发实例状态」，而不是外部网络突变，也不主要是「开发版包 vs 体验版包」身份差异。**

### 现象对照

| 场景 | 结果 | 含义 |
|------|------|------|
| 正式版（前天晚上发布） | 正常 | 线上封包 + 相对干净的使用形态 |
| 昨晚稳定体验版 | 正常 | 上传包可用；用户从微信列表打开 |
| 开发版：**刚扫码、立刻真机调试** | 正常 | 进程/会话尚干净，socket 与录音器未堆满脏状态 |
| 开发版：**几小时前扫码，调试已断开** | 实时 + 亲子均挂 | **同一份开发实例放太久，状态腐烂** |

### 机制解释（通俗）

真机调试「断开」只拔掉电脑与手机的调试线，**不会**自动：

- 关闭所有未清理的 WebSocket / SocketTask  
- 清掉 `recorderState.active` 残留  
- 卸载后台 Tab 仍挂着的实时 ASR hook  
- 回收失败按住说话留下的 `socketTaskId` 堆叠  

因此：

```text
刚扫码  → 像新车出发，状态干净 → 语音正常
几小时前扫码、调试早断 → 同一份小程序在微信里「烂了几小时」→ 再按必超时
```

这与开放社区「iOS WebSocket Connection refused，杀微信进程恢复」「socketTaskId 堆高」类报告一致；也与我们排查期 **connectSocket 双开烧签名 / 泄漏未监听 SocketTask** 完全同族。

### 对「昨天真实根因」的修正

| 旧推断（应降级） | 新推断（应升级） |
|----------------|----------------|
| 下午全员挂 = 讯飞/运营商封微信 | **状态型故障**：连接/录音生命周期清理不完整，坏状态累积 |
| 开发版挂 = 本地 dist 与体验版包不一致（主因） | **实例存活时间 + 失败重试次数** 更关键；包不一致可能是次因 |
| 真机调试断开 = 与体验版等价 | **错误**；断调试 ≠ 清状态，几小时前扫码仍会挂 |
| 亲子自愈 = 外部云恢复 | **争用/脏状态碰巧被清掉**（重扫、杀进程、页面卸载） |

### 一句话根因（截至 2026-07-15，机制层）

> **昨天语音故障的真实机制，更像是「实时 WebSocket + 全局录音器在失败后没有可靠回收，坏状态在微信进程里累积」；刚扫码/杀进程/重新打开会暂时恢复，几小时不清理的开发预览会稳定复现。下午第一次全员挂与排查期 connectSocket 双开，属于同一家族问题的不同严重度。**

### 不改代码的验证步骤（传承用）

1. 用「几小时前会挂」的那份开发版先测 → 应失败  
2. 删除该开发版 + **杀掉微信进程**  
3. 重新编译 → **立刻扫码**真机调试 → 应成功  
4. 不杀微信，放后台数小时，期间多次失败按住说话  
5. 再测亲子录音 → 若再次挂，而步骤 3 不挂 → **坐实状态腐烂**  
6. 对比：从微信列表重新打开**昨晚体验版**（非几小时前开发预览）是否仍稳  

### 排障时必须先问用户

- 是**刚扫码**还是**几小时前扫的开发版**？  
- 真机调试是**连着**还是**已断开**？（断开不代表状态已清）  
- 测的是**正式/体验**还是**开发预览**？  
- 失败后有没有**杀过微信进程**再试？（杀进程恢复 = 强证据指向客户端状态泄漏，不是甩锅用户）

## 2. 亲子录音文件：失败原因、试错、教训

### 2.1 现象

- 录完结束 → UI「上传失败，请检查网络后重试。」
- 真机调试 Network：**没有** `dialogue-transcribe` / `uploadFile` 请求记录
- 后半夜：**突然又能用**，三端都能用；再挂过、又好过（非单次随机，而是整段状态切换感）

### 2.2 根因（分层）

#### A. 错误信息被代码吞掉（确定性、已确认）

`dialogue/index.tsx` catch 原逻辑近似：

```ts
e instanceof Error ? e.message : '上传失败，请检查网络后重试。'
```

微信 `uploadFile` 失败常见形态是 **普通对象** `{ errMsg: "uploadFile:fail …" }`，**不是** `Error`。  
→ 几乎所有客户端失败都被显示成「网络失败」。  
→ Agent / 用户都无法区分：域名、文件不存在、中断、TLS、业务错误。

**这是第一层 bug：可观测性被毁。**

#### B. Network 无 URL = 请求未离开客户端（确定性推断）

若 DevTools / 真机调试 Network 完全没有上传条目，则：

- **不是** BFF 超时、nginx `proxy_read_timeout`、PM2 重启、腾讯云 ASR 挂掉（那些都会先有 HTTP 请求）
- **是** `uploadFile` 在客户端预检失败，或根本没进到真正发请求的路径

#### C. 最可能的业务根因：全局录音器争用（高置信、与「时好时坏」一致）

亲子页用 `useTapFileRecorder`；交流/预演 Tab 仍挂着 `useTencentAsrInput`（含 `navigateTo` 后隐藏存活）。

共享：

- 单一 `RecorderManager`
- `recorderState.active` + `claimRecorder` / `isMine()`

高风险路径（`d0fc2c4` 引入的 takeover）：

1. 实时 ASR 把 `active=true`，随后失权 `stopRecorder` **提前 return 且未清 `active`**
2. 亲子 `start()` 见 `active` → `waitForRecorderIdle()` → 对「其实已空闲」的 recorder 调 `stop()`
3. 触发用户铁律里写过的幽灵 `onError` / 会话损坏 → 临时 mp3 无效或路径诡异
4. `uploadFile` 客户端失败 → 无 Network → 泛化「网络失败」
5. 当实时 ASR 不再争用（超时风暴停、页面状态变）→ **突然又能用**

**这解释「整段坏 / 整段好」，不是「单次网络抖动」。**

#### D. Jul 14 下午业务 commit 未直接改亲子页

Git：`dialogue` / `useTapFileRecorder` / `recorderState` 在 `d0fc2c4` 之后到挂掉前 **无 diff**。  
→ 「下午挂」更像是 **潜伏竞态被当天更频繁的实时 ASR 失败路径打出来**，而不是亲子页被改坏。  
→ 排查时若只搜「今天下午谁改了 dialogue」会得出错误的「无人改所以是外部」—— intermediate 错误。

### 2.3 试过什么、为什么没「修掉」

| 尝试 | 意图 | 为何不算根治 |
|------|------|----------------|
| 透出 `errMsg` + `console.error` | 恢复可观测性 | **正确且必要**，但是诊断网，不是根治 |
| 上传前 `getFileInfo` | 区分「文件层」vs「upload 层」 | 正确；用户自愈后未再拿到稳定 errMsg |
| 怀疑 uploadFile 合法域名空白 | 平台配置 | 用户指出昨天也能用；且无 Network 时域名拦截未必是主因；**未证实就当主因会浪费时间** |
| 怀疑 BFF/nginx | 服务端 | **无 Network 即可否决**；本应更早否决 |
| 未改 `recorderState` 争用硬化 | 语音锁定 + 亲子已自愈 | 根因级修复**仍欠账**；自愈 ≠ 修好 |

### 2.4 亲子链路：必须记住的结论

1. 泛化「网络失败」+ 无 Network → **先修日志，再谈根因**。  
2. 与实时 ASR **共享录音器** → 实时挂掉的副作用可以表现为亲子上传挂掉。  
3. 「突然好了」优先当 **竞态条件消失**，不要当「外部服务恢复」故事讲完就收工。  
4. 根治应对齐：失权时清 `active`、禁止对空闲 recorder `stop()`、上传前 `getFileInfo`、永远透出 `errMsg`。

## 3. 实时对话（按住说话）：失败原因、试错、教训

### 3.1 现象分层（必须分开）

**原始失效（用户报告，约 Jul 14 下午起）**

- 按住 → 「语音连接超时」
- Network 可见 `/api/asr/iflytek/url?nonce=…` **200**
- 真机调试 Network **看不到** WebSocket（后来证实这是工具常态，不是「没连」的证据）

**排查期二次失效（Agent 改连接代码之后）**

- Console：`[asr-socket] connect start wss://…` → `… registering listeners` → **`onError { errMsg: "Connection refused" }`**
- **直连讯飞**与 **本站 nginx 代理** 均 Connection refused
- 同机 HTTPS 到 `yujian.yihe.site` 正常（含反复打 url 接口）

若不区分这两层，会把「我引入的回归」误当成「原始根因」，或反过来用原始现象为回归辩解。

### 3.2 已证实的事实（探针）

| 事实 | 含义 |
|------|------|
| 服务器用真实签名直连讯飞 WSS → `101 Switching Protocols` | 讯飞服务、签名算法、密钥、机房出口 **正常** |
| BFF `/api/asr/iflytek/url` 真机 200 | 签名下发 **正常** |
| 代理 location 在；curl 走代理也可 101 | nginx 代理链路 **本身可通** |
| Jul 14 业务 commit **未改** ASR 核心文件 | 「下午挂」≠「下午有人提交了 ASR」 |
| 真机调试 Size 0B / 无 WS 行 | **工具限制**，不能当 body 空 / 没发 WSS 的证据（官方 remote-debug 已知限制） |

### 3.3 排查期引入的致命代码 bug（已定位）

错误写法（概念上）：

```text
Taro.connectSocket(url)     // ① 内部已经 wx.connectSocket 一次（返回 Promise）
→ isSocketTask? false
→ nativeWx.connectSocket(url)  // ② 同一签名 URL 再连一次
```

后果：

1. **讯飞签名单次有效** → 第二次连接极易直接失败（表现为 `Connection refused` 等）  
2. 第一次连接的 SocketTask **未挂监听** → 泄漏  
3. 用户连按多次 → iOS 上 socket 状态堆坏 → **整段** WSS 全拒（直连/代理都拒）  
4. 同机 HTTPS 仍通 → 看起来像「只有 WSS 挂」，易被误判成「讯飞/运营商封了 WSS」

用户日志里的 `native sync SocketTask` 就是这条错误路径的指纹。

**正确约束**：全链路 **只调用一次** `connectSocket`；拿到 SocketTask 后 **同一调用栈** 注册全部监听。

### 3.4 原始「超时」可能是什么（诚实区间）

在加 `[asr-socket]` 日志之前，原始失效更像：

- url 200 之后握手未完成 → 客户端 8s `handshakeTimer` → 「语音连接超时」
- 可能因素包括（**按优先级，代码优先**）：
  1. `await Taro.connectSocket` 后再 `onOpen` → 官方警告的漏通知（竞态；更偏「偶发」，对「整段坏」解释力弱，用户已纠正）
  2. 与录音 claim / cleanup / 多 Tab hook 的会话 gen 交错
  3. 签名缓存（35030）——曾用 nonce + Cache-Control 防；**单独不足以解释整段挂**
  4. 排查引入的双开 / 泄漏放大成 Connection refused（**确定是代码**）

**禁止**把「今天下午开始」主叙事写成「讯飞突然封微信 4G 出口」——用户明确禁止，且直连+代理同拒 + HTTPS 通已否决该类故事。

### 3.5 试过什么、为什么没解决（甚至加重）

| 尝试 | 当时理由 | 实际结果 | 教训 |
|------|----------|----------|------|
| 只读排查（授权前） | 语音锁定 | 正确遵守流程，但后续授权后方向仍偏 | 锁定 ≠ 可以乱猜外部 |
| nonce + Cache-Control | 签名单次 / 缓存 | 未消除超时 | 局部正确，不能当银弹 |
| nginx 同 host 代理 | 绕开第三方 WSS | 当时仍超时；后有 Connection refused | 未先证明客户端 WSS 行为；且用「外部不稳」叙事 |
| 回退直连 | 怀疑同 host | 仍 refused | 同一客户端 bug 对两 host 对称 |
| 同步 `onOpen` + native | 官方要求同步监听 | **方向对**，但实现变成 Taro+native **双开** | 半吊子「按文档修」比不修更危险 |
| 再开代理 | 见 refused 以为出口问题 | 代理也 refused | **对称失败应立刻否决出口假说** |
| `[asr-socket]` 日志 | 可观测性 | **唯一真正推进证据链的手段** | 应更早、更强制要求 Console 日志 |
| 修双开（只一次 connect） | 日志指纹 | 针对排查回归的正确修复 | 需杀微信清泄漏状态再验 |

### 3.6 实时链路：必须记住的结论

1. **url 200 ≠ 语音可用**；验收点是 `onOpen` / 识别出字。  
2. **对称失败**（两 host 同 errMsg）→ 查客户端调用，不查「谁封了谁」。  
3. Taro Promise API 与 `wx.connectSocket` **不是**「再试一次 fallback」关系，而是「内部已经连过」。  
4. 排查期改动必须假设自己会引入回归；每次改完用 **新日志指纹** 验收，不要用旧现象叙事。  
5. 真机调试 Network 对 WS / Size 的表现不可靠；**Console 业务日志 > Network 面板直觉**。

## 4. 为什么之前没有发现（深层反思）

### 4.1 方法错误：用「外部故事」填「时间线空缺」

- 观察到：ASR 文件下午无 commit + 下午开始挂  
- 错误跳跃：所以是讯飞/微信/运营商变了  
- 正确做法：无 commit ≠ 无代码问题；应查 **潜伏竞态、构建产物、排查引入、共享状态、可观测性缺失**

用户多次说「昨天能用否决你的结论」——这是否定 **静态配置类** 假说（域名没配等），**不是**邀请你改讲「动态外部故障」故事。

### 4.2 证据工具用错：把 DevTools 假象当物理事实

- Size 0B → 误想 body 空（官方：真机调试可 0）  
- 无 WS 行 → 误想没 connect（真机调试常不显示 WS）  
- 应用服务端 access log / 自打 Console 日志 才能定性

### 4.3 「修一半文档」制造了更严重的 bug

- 读到「必须同步注册 onOpen」→ 上 native 同步监听  
- **没读清** Taro.connectSocket 已内部 connect  
- 加成「Taro 不行再 native」→ **双开 + 烧签名**  
- 文档合规的自我感觉，掩盖了调用次数不变量被破坏

### 4.4 两条链路绑在一起讲

- 亲子无 Network、实时 url 200——本是不同失败面  
- 合成「语音全挂 = 一个外部原因」→ 代理/直连来回，浪费大量轮次  
- 亲子自愈时仍应用同一外部故事，错过「争用消失」解释

### 4.5 可观测性债务

- 亲子吞 `errMsg`  
- 实时 initially 无 `[asr-socket]`  
- 没有日志时，任何「根因」都是文学

### 4.6 情绪与确认偏误

- 服务器 curl 101 成功 → 过度解读为「所以一定是微信到讯飞的出口」  
- 忽略对称失败与 HTTPS 对照实验  
- 用户已怒斥「别找意外概率极小的理由」后，仍短暂滑向外部叙事——**违反用户铁律**

### 4.7 对「整段坏 / 整段好」的模型错误

- 错用「单次 race / 负载时序」解释用户说的 **整段可用 vs 整段不可用**  
- 用户纠正后仍不够快地切换到：**状态机/泄漏/争用残留** 模型（双开泄漏、active 残留更贴）

## 5. 用户多次纠正思考方向时，Agent 实际如何应对（如实写）

| 用户纠正 | Agent 当时常见应对 | 问题 |
|----------|-------------------|------|
| 「别改代码，先查；域名网络没问题」 | 进入只读；但仍夹带环境怀疑 | 遵守锁定，但心态仍偏外部 |
| 「昨天能用否决你的结论」 | 口头承认，下一假设换成「下午外部变了」 | **表面服从、结构未变**（仍在系统外找变化） |
| 「是不是共享 BFF」→「不可能昨天没问题」 | 缩回 | 应系统化：共享 BFF 假说的否决条件表 |
| 「console 只有 url」 | 猜 WSS 没发出 / 代理 | 未先查真机调试 WS 是否可见 |
| 「亲子无 URL、时好时坏」 | 加 errMsg；仍偏网络文案 | 正确加诊断，但争用根治拖延 |
| 「别用时序 race 解释整段」 | 改讲连接复用整段模型，仍偏机制猜测 | 未立刻回到「我改过的代码」审计 |
| 「先上网查 DevTools 字段」 | 查了并认错 Size/WS | **本应在决策前做**，用户骂了才做 |
| 「Connection refused 别甩讯飞封 IP；改回直连；写进记忆」 | 改回直连 + 写规则；但中间又开过代理 | 对称失败后应 **禁止** 再讲出口故事 |
| 「立刻写详细经验传承」 | 本文 | 迟到的制度化 |

**模式总结**：Agent 多次在「承认用户正确」之后，**用一个新的外部故事替换旧的外部故事**，而不是把搜索空间锁死在：

1. 本会话改动的 diff  
2. 共享可变状态（recorder claim、socket 次数、签名次数）  
3. 可观测性是否被吞  
4. 对照实验（直连 vs 代理、HTTPS vs WSS、有无 Network）

这是本次浪费大量时间的主因。

### 6.2 对照实验矩阵（写下来再动手）

| 实验 | 若 A | 若 B |
|------|------|------|
| 直连 WSS vs 本站代理 WSS | 仅一方挂 → 该 host/路径 | **两方同 errMsg** → 客户端调用 |

## 7. 经验摘要（短版，可贴 HANDOFF）

1. **实时 ≠ 亲子**：拆开排；共享的是录音器，不是识别云。  
2. **无 Network 的上传失败不是 BFF**；先 `errMsg`。  
3. **亲子时好时坏 ≈ claim/active 争用**，不是玄学网络。  
4. **url 200 不算语音好了**；看 `onOpen`。  
5. **Taro.connectSocket 已经连过**；再 native = 双开烧签名 → Connection refused。  
6. **两 host 同错 = 客户端**；不要讲出口封锁。  
7. **真机调试 Network 会骗你**（0B、无 WS）。  
8. **排查改动会制造第二现场**；日志要能区分「原始病」和「医源性」。  
9. **用户纠正后禁止换皮外部假说**；锁死代码与状态。  
10. **可观测性 > 聪明假说**；没有日志不准结案。
11. **刚扫码 OK、几小时前开发预览必挂** = 状态腐烂最强旁证；真机调试断开 ≠ 清状态。
12. **正式/体验版稳、仅旧开发预览挂** ≠ 线上坏了；勿用开发态长时间实例判生产。
